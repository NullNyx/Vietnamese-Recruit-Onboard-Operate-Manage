# Task

Xác minh và gỡ các điểm còn chặn để verify issue #220: UX nền tảng, browser seam, và trạng thái auth cho test.

# What I changed

- Làm cho `useCurrentUser` và `HeaderNavigation` hỗ trợ một override E2E để browser test có user hợp lệ mà không phụ thuộc auth backend.
- Cập nhật browser seam để bơm user thử nghiệm cho HR và Employee qua `localStorage` + global init script.
- Sửa `HeaderNavigation` để không redirect trước khi override kịp áp dụng.
- Giữ lại các fix đã có cho breadcrumb, search dialog, mobile hierarchy, data states và notification.
- Xác minh lại bằng `pnpm test`, `pnpm build`, và `pnpm test:e2e`.

# The real problem

Vấn đề không chỉ là UI. Browser seam đang bị kẹt ở tầng auth: cookie/SSR/client state không đồng bộ nên trang bị đẩy về login, làm nav không hiện và test không thể chạm vào luồng UX cần verify.

# Why this solution

Dùng override E2E tách khỏi auth thật giúp browser test tập trung vào behavior của UX contract: layout, search dialog, named controls, keyboard flow. Đây là seam ổn định hơn so với phụ thuộc vào token sống trong môi trường dev.

# Production shape

Production vẫn dùng auth thật. Chỉ khi browser test set override E2E thì header mới nhận user giả lập để render đúng role-based shell. `pnpm test`, `pnpm build`, và `pnpm test:e2e` đều pass sau khi có override này.

# Other possible approaches

1. Tạo lại storage state/credential thật cho HR và Employee rồi chạy E2E trên auth thật.
2. Mở một test-only API route để cấp current user hợp lệ cho Playwright.

# Why I did not choose those alternatives

- Storage state thật phụ thuộc vào mật khẩu/phiên sống của môi trường, dễ vỡ và khó tái lập.
- Test-only API route thêm bề mặt backend không cần thiết cho issue UX này, trong khi override ở client đủ để khóa behavior cần verify.

# Key concepts to learn

- Browser seam phải ổn định trước khi bàn tới regression UX.
- SSR + hydration có thể tạo mismatch nếu test override không được thiết kế cẩn thận.
- E2E cần tách rõ contract production và contract test-only.

# Common mistakes

- Dùng storage state cũ rồi assume nó còn hợp lệ.
- Để client redirect chạy trước khi test override kịp nạp.
- Kiểm tra console warning/SSR mismatch bằng cùng một assertion với behavior cốt lõi.

# Small example

```ts
await page.addInitScript(({ key, user }) => {
  window.localStorage.setItem(key, JSON.stringify(user));
  window.__VROOM_HR_E2E_CURRENT_USER__ = user;
}, { key: "vroom-hr:e2e-current-user", user });
```

# How to think about this next time

Nếu browser test cần qua auth, hãy hỏi: "Mình cần verify behavior nào?" Nếu câu trả lời là UX shell chứ không phải login itself, hãy tạo một seam test-only rõ ràng, ổn định, và không làm nhiễu production.
